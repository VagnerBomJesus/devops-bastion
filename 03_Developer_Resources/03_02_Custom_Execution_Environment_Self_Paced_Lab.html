1. Start Lab Environment

You need to run following steps if your lab environment has shut down. To conserve resources, the lab environment shuts down automatically after eight hours.

1.1. Start Environment After Shut Down

  1. Go to the OPENTLC lab portal and use your OPENTLC credentials to log in.

  2. Navigate to Services → My Services (this should be the screen shown right after logging in).

  3. In the list of your services, select your lab environment.

  4. Select App Control → Start to start your lab environment.

  5. Select Yes at the Are you sure? prompt.

  6. On the bottom right, click Submit.

    After a few minutes, expect to receive an email letting you know that the lab environment has been started.

1.2. Connect bastion Node

  1. Bastion node is the main host which is accssible via ssh from public network. You will be using this host for running commands and accessing other host available in your environment. You will use ssh to connect bastion node, using the login details provided in third email.

    [laptop]$ ssh <OPENTLC-USER-ID>@bastion.${GUID}.${SUBDOMAIN}

  2. Switch to devops user. The User devops is service account in bastion node who has complete sudoer privilege to run all commands with no sudo password.

    [bastion]$ sudo su - devops

1.3. Refresh Satellite Manifest

  1. After restart of your classroom environment need to run following Lab Manager command to refresh satellite manifest.

    $ lab refresh -i satellite

You need to perform all of the lab instruction on bastion node as devops user.

3. Installing Ansible Builder

In this part of the lab, You will install ansible-builder to build container image. Ansible Builder is a command line tool that automates the process of building automation execution environments.

  1. You can install Ansible Builder using Red Hat Subscription Management to register and attach to your Red Hat Ansible Automation Platform subscription. For your convenience node have all necessary repositories enabled. You can directly run dnf cli to install ansible-builder package.

    $ sudo dnf install -y ansible-builder
$ ansible-builder --version
    Version output
    1.0.1

4. Building Definition File

4.1. Definition file

Ansible Builder usage definition file to create execution environment image. Ansible Builder reads and validates definition file, then create a Containerfile, and finally pass the Containerfile to Podman which then packages and creates automation execution environment image.

  1. By default, Ansible Builder will look for a definition file named execution-environment.yml. So you will create a default definition file including all dependencies as per the lab requirements.

    $ mkdir $HOME/ee-osp-build
$ cd $HOME/ee-osp-build
$ cat > execution-environment.yml <<EOF
version: 1
dependencies:
  galaxy: requirements.yml
  python: requirements.txt
  system: bindep.txt
EOF

    • galaxy: will install collections.

    • python: will install python libraries.

    • system: will install rpm packages.

4.2. Galaxy Dependencies

The galaxy entry in the dependenices section of definition file, points to valid requirements.yml file for the ansible-galaxy collection install -r command. The requirements.yml is a file that installs ansible collections. It may be listed as a relative path or an absolute path.

  1. You need to create galaxy requirements.yml in $HOME/ee-osp-build directory as per lab collection requirements.

    $ cd $HOME/ee-osp-build
$ cat > requirements.yml <<EOF
collections:
  - openstack.cloud
  - community.general
  - ansible.posix
EOF

4.3. Python Dependencies

The python entry in the dependenices section of definition file, points to valid requirements.txt file for the pip install -r command. The requirements.txt is a file that installs extra Python requirements on top of what the Collections already list as their Python dependencies. It may be listed as a relative path or an absolute path.

  1. You need to create python requirements.txt in $HOME/ee-osp-build directory as per lab python library requirements.

    $ cd $HOME/ee-osp-build
$ cat > requirements.txt <<EOF
openstacksdk==0.36.5
EOF

4.4. System Dependencies

The system entry in the dependenices section of definition file, points to a bindep requirements file, which will install system-level dependencies that are outside of what the collections already include as their dependencies. It may be listed as a relative path or an absolute path.

  1. You need to system bindep.txt in $HOME/ee-osp-build directory as per lab system packages requirements.

    $ cd $HOME/ee-osp-build
$ cat > bindep.txt <<EOF
python3 [platform:rpm]
EOF

6. Test Custom Execution Environment

In this part of the lab you will test custom execution environment image. You will write a playbook which will be using openstack.cloud collection that you will run against default image and the custom image (created in previous steps) to verify that custom image serves the need of the lab requirement.

6.1. Create Playbook & Navigator settings file

  1. You need to create following playbook to test images as follows.

    $ mkdir $HOME/test-osp-image
$ cd $HOME/test-osp-image
$ cat > playbook.yml <<EOF
---
- name: Test Custom EE
  hosts: localhost
  gather_facts: false
  vars:
    guid: "{{ lookup('env', 'GUID')}}"
  tasks:
    - name: Collect OpenStack instances info
      openstack.cloud.server_info:
        cloud: "{{ guid }}-project"
        server: "*"
      register: output

    - name: Print output
      debug:
        var: output
EOF

  2. Create ansible-navigator settings file as follows.

    $ cat > ansible-navigator.yaml <<EOF
ansible-navigator:
  ansible:
    playbook: playbook.yml
  app: run
  execution-environment:
    pull-policy: missing
    volume-mounts:
      - src: "/etc/openstack/clouds.yaml"
        dest: "/etc/openstack/clouds.yaml"
        label: "Z"
    environment-variables:
      pass:
        - GUID
  mode: stdout
  playbook-artifact:
    enable: False
EOF

6.2. Run Against Default Execution Environment

  1. You can simply execute ansible-navigator to run playbook with default execution environment.

    $ ansible-navigator

7.1. Tag New Execution Environment Image

  1. An image name is made up of slash-separated name components, optionally prefixed by a registry hostname. The hostname must comply with standard DNS rules. If a hostname is present, it may optionally be followed by a port number in the format :5000. You run following steps to tag custom execution environment.

    $ podman tag localhost/ee-osp:1.0.0 bastion.example.com:5000/custom/ee-osp:1.0.0

7.2. Push Execution Environment Image

  1. Now, you can push custom image to a registry as follows. Just for info, we have configured local secure registry using self signed certificate so you can use --tls-verify=false to ignore certificate validation.

    $ podman login bastion.example.com:5000 -u admin -p r3dh4t1! --tls-verify=false
$ podman push bastion.example.com:5000/custom/ee-osp:1.0.0 --tls-verify=false

7.3. Search Image In Registry

  1. You can run following command to search image in registry.

    $ podman search bastion.example.com:5000/custom --tls-verify=false
