3. Introduction of Git & GitLab

A quick description: git and GitLab are not the same thing. Git is an open-source, version control tool; GitLab is an opensource company which develops software for the software development lifecycle which integrate with git. You do not need Gitlab to use git, but you cannot use GitLab without using git. There are many other alternatives to GitLab, such as GitHub, BitBucket, and “host-your-own” solutions such as gogs and gittea. All of these are referred to in git-speak as “remotes”, and all are completely optional. You do not need to use a remote to use git, but it will make sharing your code with others easier.

4. Local Git Repository

A Git repository is a virtual storage which allows you to save versions of your code, which you can access when needed. In this section, you will learn how to set up a repository (repo) under Git version control. This resource will walk you through initializing a Git repository for a new or existing project.

4.1. Initialize Git Repository

In this section of the lab you will install, initialize, configure, add, and commit file to git repository.

  1. You can start with git package installation as follows.

    $ sudo dnf install git -y

  2. To initialize a git repository, You need to create a project directory in which you will do all your code development work.

    $ mkdir $HOME/demo-project
$ cd $HOME/demo-project

  3. Now, you need to run one-time git init initial command to initialize local git repository in the project directory.

    $ git init
$ git checkout -b main
    GitHub and GitLab renamed default master branch to main branch. More info: https://github.com/github/renaming and https://about.gitlab.com/blog/2021/03/10/new-git-default-branch-name/

4.2. Git User Info

  1. In addtion to initialization, you also need to set global git configuration options such as username or email. You will use git config command in the repository to setup. Git stores configuration in three separate files, which lets you scope options to individual repositories (local) in ./.git/config, User-specific settings (Global) in $HOME/.gitconfig, or the entire system (system) in /etc/gitconfig.

    $ git config --global user.name "devops"
$ git config --global user.email "devops@example.com"
$ git config --list
    Git config output
    user.email=devops@example.com
user.name=devops
core.repositoryformatversion=0
core.filemode=true
core.bare=false
core.logallrefupdates=true

6. Writting Content in Local Repository

In this part of lab you will write content, add, and commit conntent to the local git repository.

6.1. Create New File

  1. Create a README.adoc file in your project directory and use git add command to add a change in the working directory to the staging area. The git add command tells Git that you want to include updates to a particular file in the next commit.

    $ cat >> README.adoc <<EOF
> Welcome to Demo-Project
EOF

$ git add README.adoc

6.2. Staging Changes

  1. In conjunction with git add commands, you’ll also need git status to view the state of the working directory and the staging area. The staging area is one of Git’s more unique features. You can think of it as a buffer between the working directory and the project history. The staging area is considered one of the "three trees" of Git, along with, the working directory, and the commit history.

    $ git status
    Status output
    On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
	new file:   README.adoc

6.3. Commit Changes

  1. The git commit command captures a snapshot of the project’s currently staged changes. Committed snapshots can be thought of as “safe” versions of a project. Git will never change them unless you explicitly ask it to.

    $ git commit -m "first commit"
    Commit Output
    [main (root-commit) c96dc48] first commit
 1 file changed, 1 insertion(+)
 create mode 100644 README.adoc

8. Securing Repository Connection

You have seen in previous steps, you work locally with git and then push the commits to GitLab remote repository over the network. So communication between Git and Gitlab should be secure. To secure communication, you can use ssh protocol. You also get benefit of using SSH in authentication because after ssh key implementation, you don’t need to supply your username and password each time you push content to remote repository. In this part of the lab you will implement secure communication.

8.1. SSH keys

SSH public key authentication improves security considerably as it frees the users from remembering complicated passwords. The ssh public key encryption uses Asymmetric Cryptography algorithms, works with two separate keys. These two keys form a pair that is specific to each user.

Each SSH key pair includes two keys:

  • Public Key: This key gets copied to ssh server, In your case, GitLab Server.

  • Private Key: This key remains with the user.

    1. You can use ssh-keygen command line tool to generate ssh private and public key pair. However, we have already stored private and public key in devops user’s $HOME/.ssh directory which you will use in this lab. You only need to copy content of public key as follows-

      $ cat $HOME/.ssh/${GUID}key.pub
      Sample Output

9. Creating Branch

In Git, Branching means you diverge from the main line of development and continue to do work without messing with that main line. When you want to add a new feature or fix a bug, you spawn a new branch to encapsulate your changes.

9.1. Add New Branch

  1. You can use git checkout command to switch or create new development branch from main branch as follows-

    $ git checkout -b dev-branch

  2. You can use git branch command to list all of the existing branches.

    $ git branch
    Branch output
    * dev-branch
  main
    The branch name with the asterisk is your current branch.

9.2. Push Branch to Remote Repository

  1. To push local branch dev-branch to remote repository. You need to run following command.

    ]$ git push origin dev-branch

  2. To verify that local branch has been pushed. You can use following steps-

    3. Then click on demo-project on next page.

    4. Then click on Branches

      Figure 4. Branches Link

    5. Next page will display new branch dev-branch along with other branches.

10. Create and Merge Pull Request

Pull request is a feature that let you tell other developers about changes you’ve pushed to your remote branch, in your case Gitlab dev-branch repository. Gitlab provides a user-friendly web interface to open a pull request. Once a pull request is opened, you can discuss and review the potential changes with collaborators before your changes are merged into the main branch.

10.1. Create, Stage and Commit

In this section, you will create a simple playbook as code, then you will commit the changes in local repository.

  1. You can follow the steps to create main.yml playbook in local git dev-branch.

    $ git checkout dev-branch
$ cat >> main.yml <<EOF
---
- name: Play for PR
  hosts: localhost
  tasks:
    - name: Welcome message
      debug:
        msg: "PR is merged"
EOF

10.5. Pull Content

Git pull updates your current local working branch, and all of the remote tracking branches. Without running git pull, your local repository will never be updated with changes from the remote. git pull should be used every day you interact with a repository with a remote, at the minimum. That’s why git pull is one of the most used Git commands.

  1. You can use git pull command to fetch updates from GitLab remote main branch repository to your local git main branch, as follows -

    $ git checkout main
$ ls -l
README.adoc
$ git pull
